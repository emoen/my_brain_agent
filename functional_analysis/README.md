# Functional Analysis — LLM Knowledge Base

A self-contained knowledge base for Functional Analysis, built following
[Andrej Karpathy's LLM Knowledge Base pattern](https://x.com/karpathy):

> *Raw source documents are collected into `raw/`, then compiled by an LLM into
> a `.md` wiki. The wiki is operated on by CLI tools for Q&A and incremental
> enhancement. All of it is viewable in Obsidian. You rarely write or edit the
> wiki manually — it is the domain of the LLM.*

---

## Folder Structure

```
functional_analysis/
├── raw/                        ← Source documents (PDFs, papers, notes)
│   └── *.pdf
├── wiki/                       ← Auto-generated by ingest_knowledge_base.py
│   └── *.md                       Obsidian-ready markdown articles
├── ingest_knowledge_base.py    ← Step 1: PDF → wiki (this script)
└── README.md                   ← This file
```

## Step 1 — Ingestion (`ingest_knowledge_base.py`)

Reads every PDF in `raw/`, extracts full text, and uses **gpt-4o** via the
GitHub Models API to produce a structured wiki article for each one.

### What each generated article contains

| Section | Description |
|---|---|
| `## Overview` | 2–4 paragraph summary of scope and significance |
| `## Key Concepts` | Bullet list of definitions, theorems, and results |
| `## Chapter / Section Map` | Outline of the document structure |
| `## Connections & Backlinks` | `[[Wikilinks]]` to related concepts (Obsidian graph edges) |
| `## Open Questions & Further Reading` | Gaps raised and related resources |

### Features

- **Incremental** — skips PDFs that already have a wiki entry; safe to re-run
- **Chunked** — text is split into conservative 12k-char chunks to fit this endpoint’s 16k request-body cap for gpt-4o
- **Hierarchical merge** — chunk summaries are merged in batches to avoid overflow during final synthesis
- **Obsidian-ready** — YAML frontmatter + `[[wikilinks]]` for graph view
- **Rate-aware** — gpt-4o has 15 req/min, 150 req/day on Copilot Business

### Running

```bash
# from project root
cd /home/endrem/project/my_brain
source .venv/bin/activate

python functional_analysis/ingest_knowledge_base.py           # ingest new PDFs
python functional_analysis/ingest_knowledge_base.py --force   # re-process all
```

Or from inside the topic directory:

```bash
cd functional_analysis
source ../.venv/bin/activate
python ingest_knowledge_base.py
```

### Requirements

- Python venv at `../.venv` with `openai`, `httpx`, `pypdf` installed
- `TOKEN` in `ingest_knowledge_base.py` set to a GitHub PAT with `models:read`
- Source PDFs placed in `raw/`

### Troubleshooting

**Error: "Request body too large"**
- This endpoint currently enforces a 16k-token max request body for gpt-4o
- Script now uses 12k-char chunks plus batched (hierarchical) merges
- Run without `--force` to process only PDFs that do not yet have `.md` output

## Step 1B — Chapter-Level Ingestion (`ingest_chapters.py`)

Generates one **rich markdown file per chapter** in:

- `wiki/chapters/<book_slug>/chapter_XX_<title>.md`

### Prompt behavior (strict extraction)

The chapter pipeline now explicitly forces:

- **Definitions & Notation** (explicit terms/symbols)
- **Theorems, Lemmas & Corollaries** (full statements with hypotheses/conclusions)
- **Proof Ideas** (for each major result when present)
- **Worked Examples**, **Exercises**, and **Key Insights**
- No invented facts beyond extracted chapter text/notes

### Running

```bash
cd /home/endrem/project/my_brain/functional_analysis

# Recommended: auto-activate env and auto model fallback
bash run_ingest_chapters.sh

# Incremental run (tries gpt-4o first, falls back to gpt-4o-mini if quota is exhausted)
python ingest_chapters.py --model auto

# Force re-read and regenerate ALL chapters
python ingest_chapters.py --model auto --force
```

### Model fallback behavior

- `--model auto` (default) probes `gpt-4o` first.
- If `gpt-4o` is unavailable or daily quota/rate limit is exhausted, it automatically falls back to `gpt-4o-mini`.
- Chunk size and merge batch settings are chosen per selected model to reduce token-limit failures.

### Environment activation helper

- `run_ingest_chapters.sh` checks whether a virtual environment is active.
- If none is active, it automatically sources `../.venv/bin/activate` and then runs `ingest_chapters.py`.

## Theorem Connection Graph

You can automatically build a graph showing how theorem-like results reference each other across chapter notes.

### Run

```bash
cd /home/endrem/project/my_brain/functional_analysis
source ../.venv/bin/activate

python build_theorem_graph.py

# Strict dependency graph (numbered references only)
python build_theorem_graph.py --mode dependency

# Logical dependency graph (numbered references + theorem-title mentions)
python build_theorem_graph.py --mode logical
```

This generates:

- `wiki/graphs/theorem_graph.dot` (Graphviz)
- `wiki/graphs/theorem_graph.mmd` (Mermaid)

### Optional rendering with Graphviz

```bash
dot -Tpng wiki/graphs/theorem_graph.dot -o wiki/graphs/theorem_graph.png
```
